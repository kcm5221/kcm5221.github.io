1. 프로젝트 개요

RunTimer는 인터벌 러닝 훈련에 최적화된 iOS·Android 크로스플랫폼 헬스 앱입니다.
타이머 · 스케줄 알람 · GPS 거리측정을 하나의 화면 흐름으로 통합하여 초보 러너부터 마라토너까지 손쉽게 훈련 계획을 세우고 기록할 수 있도록 돕습니다.

  • 인터벌 타이머 – 워밍업·러닝·워킹·쿨다운을 자유롭게 편집하고 반복 세트를 저장
  • 스케줄 알람 – 요일/시간 기반 알림으로 훈련 루틴을 습관화
  • 실시간 GPS 트래킹 – 거리, 페이스, 고도, 칼로리를 자동 계산
  • 히스토리 & 통계 – 달성률, 누적 거리, 평균 페이스를 시각화
  • 백그라운드 실행 – 화면이 꺼져도 음성·진동·알림으로 세트 진행 상황 안내

RunTimer 이름 선정 이유

영어권 사용자가 즉시 의미를 유추할 수 있는 단어 조합을 선택했습니다.
Run + Timer 두 단어만으로 앱의 핵심 기능(달리기 & 타이머)을 곧바로 전달하며, 철자·발음이 짧아 기억하기 쉽습니다. 또한 App Store · Play 검색에서 러닝/타이머 키워드와 자연스럽게 매칭되어 ASO(앱스토어 최적화)에 유리합니다.

2. Flutter vs React Native (선택 이유)

  • 백그라운드 타이머·알람 안정성 – Flutter는 flutter_local_notifications + workmanager로 iOS/Android 모두 동일 코드·높은 신뢰도. React Native는 JS 스레드가 백그라운드에서 중단되기 쉬워 iOS에서 알람 지연/누락 사례가 잦음.
  • 성능 & UI 일관성 – Flutter는 Skia 엔진으로 자체 렌더링 → 애니메이션·지도 스크롤이 네이티브급. React Native는 JS↔Native 브리지로 프레임 드랍 가능성과 플랫폼별 UI 편차 존재.
  • GPS & 센서 패키지 품질 – Flutter의 geolocator, permission_handler는 활발히 유지관리·문서화. React Native는 모듈 난립·권한 설정 이슈로 새 OS 버전 대응이 느림.
  • 빌드/배포 효율 – Flutter는 단일 flutter build 명령과 GitHub Actions에서 두 플랫폼 동시 아카이브. React Native는 Android Gradle + Xcode/Cocoapods 분리 관리 → CI 설정·버전 충돌 부담.
  • 안정성 & 디버깅 – Flutter는 Null‑safety 기반 단일 언어(Dart)로 런타임 오류가 적고, DevTools Performance Charts 지원. React Native는 JS/Native 두 스택을 모두 추적해야 해 크래시 루트 분석이 복잡.

러닝 앱은 실시간 처리 + 백그라운드 안정성 + 하드웨어 센서가 핵심입니다. 위 비교처럼 해당 요구 조건에 더 적합하고, 유지보수와 배포 효율까지 고려했을 때 Flutter가 RunTimer의 기술 스택으로 최적이라고 판단했습니다.

3. 핵심 구현

  • 타이머 엔진 – 고정/가변 인터벌, 라운드·세트·휴식 계산
  • 스케줄 알람 매니저 – 백그라운드에서 요일별 푸시 예약
  • GPS 서비스 – 위치 스트림으로 거리·페이스·코스 기록
  • 데이터 저장 – Hive 기반 로컬 DB + Google Drive 백업(예정)
  • 통계 대시보드 – 누적 거리, 월간 그래프, 최고 기록 하이라이트

4. 문제 해결 사례

  • iOS 백그라운드 제한 – BGTaskScheduler + WorkManager로 알람 누락 0%
  • GPS 드리프트 – Kalman 필터로 오차 ±3 m로 감소
  • 배터리 과소비 – 위치 업데이트 간격을 자동 조절하여 사용량 25%↓
  • 알람 중복 – ID 해시 기반 예약, 중복 발생 0%
